# 答辩与作品集材料

## 一句话项目介绍

宠伴管家是一套单门店宠物洗护预约会员系统，真实联动 C 端、商家后台、API 和 PostgreSQL，覆盖预约、订单、支付、优惠券、充值、套餐卡、通知、报表与可降级 AI。

## 最值得讲的四个设计点

1. **业务闭环完整**：预约不是终点，后续还有确认、订单、支付、优惠核销、会员权益、消费记录和通知联动。
2. **事务一致性**：余额支付、套餐卡核销和优惠券支付都在事务中完成，失败场景会验证无部分更新。
3. **时间口径可靠**：预约和报表统一使用上海时区；优惠券有效期按上海日期判断，避免 UTC 午夜边界错误。
4. **AI 与核心业务解耦**：真实模型可调用业务工具，但模型不可用时规则降级；预约草案必须由用户确认提交。

## 常见答辩问题

### 为什么没有直接做多门店 SaaS？

当前版本优先验证单门店的核心业务模型。多租户会引入 tenant 隔离、门店权限、跨店会员和配置继承，应该在单店闭环稳定后扩展，而不是提前增加复杂度。

### 为什么支付是模拟或到店支付？

真实微信/支付宝支付需要商户资质、签名、回调验签、退款和对账。当前项目重点是支付后的业务一致性，接口和状态流已经为真实渠道预留，但不虚构外部支付能力。

### AI 的价值在哪里？

AI 读取服务、宠物、可用时段、知识库和经营统计，不是孤立聊天窗口。安全边界是模型只生成草案，关键业务写操作仍由确定性 API 和用户确认完成。

### 如何证明支付不会扣一半？

余额、套餐卡、优惠券、订单、预约与消费记录在 Prisma 事务内更新；自动化测试覆盖余额不足、错误套餐卡、过期券和重复用券，并断言失败后数据库没有部分变化。

### 项目目前最大的技术债是什么？

浏览器 E2E 目前集中在一条核心成功路径；还需要补异常 UI 分支、视觉回归、真实支付沙箱、生产监控告警、RBAC 和部署环境的持续验收。

## 作品集描述

> 独立完成宠物门店预约会员管理系统，采用 Vue 3、uni-app、NestJS、Prisma 与 PostgreSQL，设计预约冲突检测、订单支付事务、优惠券/余额/套餐卡核销、站内通知和上海时区经营报表；接入可降级的 LLM 工具调用客服；建立 16 个真实数据库 API 集成测试、Playwright 跨端支付 E2E 和 GitHub Actions CI。
