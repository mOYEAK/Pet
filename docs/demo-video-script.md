# 演示视频脚本

建议时长：9 分钟。录制前执行 `pnpm typecheck` 和三端 build，准备好演示账号与固定浏览器标签页。

## 已准备素材

- `docs/assets/showcase/admin-walkthrough.webm`：后台无旁白原始录屏，1440×900，约 21 秒。
- `docs/assets/showcase/miniapp-walkthrough.webm`：C 端无旁白原始录屏，430×932，约 11 秒。
- `docs/assets/showcase/01` 至 `08`：预约、支付、后台、会员、报表、AI 和 CI 共 8 张固定截图。
- 完整索引和复录命令见 [showcase-assets.md](showcase-assets.md)。

两段原始录屏用于视频剪辑中的快速过场，不替代下方 9 分钟旁白录制。正式成片建议使用本人的声音按时间轴讲解，这样答辩时表达与视频保持一致。

## 0:00–0:35 开场与定位

画面：README 标题、项目结构或首页。

讲解：

> 宠伴管家是一套面向单门店宠物洗护场景的预约会员管理系统。它包含 uni-app C 端、Vue 商家后台、NestJS API、Prisma/PostgreSQL，以及可降级的 AI 客服和经营分析能力。重点不是页面堆砌，而是跑通预约、订单、支付、优惠券和会员权益的真实业务闭环。

## 0:35–1:10 架构与数据边界

画面：仓库目录、`docker-compose.yml`、CI 工作流。

要点：

- monorepo 分为 C 端、后台、API、共享类型和 Agent 包。
- PostgreSQL 保存业务数据，Prisma migrations 管理结构。
- AI 不可用时规则降级，核心业务不依赖模型。

## 1:10–2:25 C 端创建预约

画面：`http://localhost:5174`。

操作：

1. 查看首页门店信息和服务入口。
2. 打开宠物档案，确认演示宠物“咪咪”。
3. 进入预约页，选择猫咪洗护、未来日期和可用时间段。
4. 填写备注并提交。
5. 展示状态为“待确认”和预约提交通知。

讲解重点：服务、宠物、时间段和预约记录均来自 API；后端会校验宠物归属、停用服务、过去时间和时段冲突。

## 2:25–4:20 后台确认与订单支付

画面：`http://localhost:5173`，管理员 `19900000000`。

操作：

1. 在预约管理找到刚提交的预约并确认。
2. 从预约生成订单。
3. 打开订单管理，选择到店支付。
4. 选择“老客户护理满减券”，确认应收 128、优惠 20、实收 108。
5. 完成支付，展示订单已支付。

讲解重点：支付事务会同时更新订单、预约、优惠券、消费记录和通知；任一校验失败都不会留下部分更新。

## 4:20–5:05 C 端结果回显

画面：回到 C 端预约页和会员中心。

操作：刷新预约，展示“已完成”“已支付”“优惠 ¥20”“实付 ¥108”；再展示消费记录和支付通知。

## 5:05–6:10 会员与套餐卡

画面：后台会员管理、C 端会员中心。

操作：

1. 展示充值的实收、赠送与余额前后值。
2. 展示套餐卡剩余次数和有效期。
3. 说明余额不足、服务不匹配、过期和零次数都会被拒绝并回滚。

## 6:10–7:15 报表与 AI

画面：经营报表、AI 经营助手、C 端智能客服。

要点：

- 报表按上海时区和实际支付时间统计。
- 套餐卡核销不重复计入现金收入。
- 客服能查询真实服务、宠物与可用时段，并只生成预约草案，不替用户提交。
- 无密钥或模型失败时自动规则降级。

## 7:15–8:20 自动化质量证据

画面：终端测试输出、GitHub Actions 页面。

讲解：

> API 有 3 组 16 个真实 PostgreSQL 集成测试，总行覆盖率 62.44%，经营统计服务行覆盖率 100%。Playwright 还会真实操作 C 端和后台，完成预约、确认、订单、优惠券支付与结果回显。CI 会持续执行覆盖率、浏览器 E2E、全仓 typecheck 和三端生产构建。

## 8:20–9:00 部署与收尾

画面：Docker/Nginx 文件、部署文档、README。

讲解：

> 项目提供 API 和 Web 生产镜像、Nginx 统一入口、正式 migrations、seed 与完整验收文档。当前定位是单门店 MVP，真实微信支付、短信、多门店 SaaS、复杂 RBAC 和向量 RAG 是明确的后续边界。

结尾：用一句话回到价值——“用完整、可验证的业务闭环，降低宠物门店预约与会员经营的人工成本。”
